# Бронювання шезлонгів біля басейну — Підгорецький Маєток

Система онлайн-бронювання шезлонгів: клієнтський флоу (`/booking`) + адмін-панель
(`/admin`). Збудовано на **Next.js 14 + Supabase**, готово до хостингу на **Vercel**.

- Клієнт обирає дату, сеанс, кількість гостей, шезлонг на карті басейну й оплачує.
- Персонал у `/admin` бачить статистику, всі бронювання, календар, робить ручні
  бронювання та блокує місця (ремонт/резерв).
- Ціни рахуються **на сервері** — браузеру не довіряємо.
- Один шезлонг неможливо забронювати двічі (атомарна перевірка в базі).

## Стек і структура

- **Next.js 14** (App Router, TypeScript) — фронтенд + серверні API-роути.
- **Supabase** (PostgreSQL + Auth) — база бронювань і вхід персоналу.
- **Vercel** — хостинг.
- **LiqPay** (ПриватБанк) — онлайн-оплата (опційно; поки порожні ключі — демо-режим).

```
app/
  booking/page.tsx          ← клієнтський флоу (5 кроків)
  admin/page.tsx            ← перевірка доступу (сервер)
  admin/AdminDashboard.tsx  ← адмінка (5 вкладок)
  admin/login/page.tsx      ← вхід персоналу
  api/                      ← серверні роути (доступність, бронювання, блок, оплата)
components/PoolMap.tsx       ← інтерактивна карта 40 шезлонгів
lib/
  pricing.ts                ← ціни (єдине джерело правди)
  spots.ts                  ← координати 40 шезлонгів
  booking-service.ts        ← валідація + атомарне створення (тільки сервер)
  supabase/                 ← клієнти Supabase + перевірка адміна
supabase/schema.sql         ← SQL-схема: таблиці, RLS, функція create_booking
middleware.ts               ← захист /admin + оновлення сесії
```

## Швидкий старт локально

Потрібен **Node.js 18.18+** (краще 20+).

```bash
npm install
cp .env.local.example .env.local   # потім заповни значення (див. нижче)
npm run dev
```

Відкрий:
- Клієнт: http://localhost:3000/booking
- Адмінка: http://localhost:3000/admin (спершу попросить увійти)

> Без заповненого `.env.local` сторінки відкриються, але бронювання й вхід не
> працюватимуть — спочатку зроби крок «Налаштування Supabase».

## Налаштування Supabase

1. Зареєструйся на https://supabase.com → **New project**. Запиши пароль бази.
2. Коли проект створиться, відкрий **SQL Editor → New query**, встав весь вміст файлу
   **`supabase/schema.sql`** і натисни **Run**. Це створить таблиці `bookings`,
   `blocked_spots`, функцію `create_booking` і політики безпеки (RLS).
3. Візьми ключі: **Settings → API**:
   - `Project URL` → `NEXT_PUBLIC_SUPABASE_URL`
   - `anon public` → `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - `service_role` (секрет!) → `SUPABASE_SERVICE_ROLE_KEY`
4. Створи акаунт адміна: **Authentication → Users → Add user → Create new user**.
   Введи email і пароль персоналу (напр. `pidgoretskyyhotel@gmail.com`). Постав
   галочку **Auto Confirm User**, щоб не треба було підтверджувати пошту.
   Цей самий email впиши у `ADMIN_EMAILS`.

> **Хто має доступ до `/admin`?** Будь-який користувач Supabase, чий email є у списку
> `ADMIN_EMAILS` (через кому). Хочеш кілька працівників — додай їм акаунти й перелічи
> їхні email. Якщо `ADMIN_EMAILS` порожній — пускає будь-якого авторизованого.

## Змінні оточення

Файл `.env.local` (локально) і ті самі змінні в налаштуваннях Vercel:

| Змінна | Обовʼязкова | Що це |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | так | URL проекту Supabase |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | так | Публічний ключ Supabase |
| `SUPABASE_SERVICE_ROLE_KEY` | так | Секретний ключ (тільки сервер!) |
| `ADMIN_EMAILS` | так | Email-и адмінів через кому |
| `NEXT_PUBLIC_SITE_URL` | для оплати | Публічна адреса (для LiqPay callback) |
| `LIQPAY_PUBLIC_KEY` | ні | Ключ LiqPay (порожній = демо) |
| `LIQPAY_PRIVATE_KEY` | ні | Секрет LiqPay (порожній = демо) |

⚠️ `SUPABASE_SERVICE_ROLE_KEY` і `LIQPAY_PRIVATE_KEY` — **ніколи** не додавай у змінні з
префіксом `NEXT_PUBLIC_`, інакше вони потраплять у браузер.

## Деплой на Vercel

1. Залий проект у GitHub (новий репозиторій).
2. На https://vercel.com → **Add New → Project** → імпортуй репозиторій.
3. У **Environment Variables** додай усі змінні з таблиці вище
   (значення `NEXT_PUBLIC_SITE_URL` — поки твій майбутній прод-домен або URL від Vercel).
4. **Deploy**. Vercel сам визначить Next.js і збере проект.

Після деплою отримаєш адресу типу `pidhoretsky-pool.vercel.app`. Перевір
`…/booking` і `…/admin`. Далі — підключення власного домену.

## Той самий домен, різні сторінки

Ти хочеш, щоб бронювання було на `www.pidgoretskyymaietok.com/booking`, а адмінка на
`/admin`, поряд з наявним сайтом. Є три способи — обери один.

### ⭐ Варіант A (рекомендований): один сайт на Vercel

Найпростіше і найнадійніше — обслуговувати весь домен з цього застосунку.

1. Поклади файли наявного статичного сайту в теку **`public/`** цього проекту:
   `index.html`, `pool.html`, `hotel.html`, `palace.html`, `events.html`,
   `contact.html` і теку `images/`. Vercel віддаватиме їх напряму:
   `public/pool.html` → `…/pool.html`, `public/images/...` → `…/images/...`.
2. **Видали файл `app/page.tsx`** — тоді корінь `/` показуватиме твій
   `public/index.html` (інакше пріоритет матиме застосунок і робитиме редірект).
3. У Vercel: **Settings → Domains** → додай `www.pidgoretskyymaietok.com`
   (і `pidgoretskyymaietok.com` з редіректом на `www`). Vercel підкаже, які
   DNS-записи поставити у реєстратора домену.

Готово: `/`, `/pool.html` тощо — твій сайт; `/booking` і `/admin` — застосунок.
Один деплой, один домен.

> На сторінці «Басейн» (`pool.html`) додай кнопку-посилання на `/booking`:
> `<a href="/booking">Забронювати шезлонг</a>`.

### Варіант B: домен на Vercel + проксі на старий хостинг

Якщо не хочеш переносити статику. Домен веде на Vercel; застосунок віддає
`/booking`, `/admin`, `/api`, а решту шляхів проксі-перенаправляє на твій
поточний хостинг.

1. Перейменуй `vercel.proxy.example.json` → `vercel.json`.
2. Встав у ньому реальний origin твоєї статики замість `https://СТАРИЙ-ХОСТ-СТАТИКИ`.
3. Підключи домен у Vercel (як у Варіанті A, крок 3).

Мінус: кожна сторінка маркетингу йде через проксі (трохи повільніше) і потрібен
сталий окремий хост для статики.

### Варіант C: піддомен (найшвидше, але інша адреса)

Замість сторінки — піддомен `booking.pidgoretskyymaietok.com`. У Vercel додаєш цей
піддомен, у реєстратора — один CNAME на Vercel. Головний сайт лишається де є.
Адреса буде не `/booking`, а піддомен — якщо тобі ок, це найменше клопоту.

## Оплата ПриватБанк (LiqPay)

Зараз працює **демо-режим**: «оплата» одразу створює підтверджене бронювання, кошти
не списуються. Це зручно для тестів і запуску.

Щоб увімкнути справжню оплату карткою (Visa/Mastercard через Приват24):

1. Зареєструй мерчанта на https://www.liqpay.ua/ (LiqPay — це платіжний сервіс
   ПриватБанку). Отримай `public_key` і `private_key`.
2. Впиши їх у змінні `LIQPAY_PUBLIC_KEY` / `LIQPAY_PRIVATE_KEY` (локально й на Vercel).
3. Встанови `NEXT_PUBLIC_SITE_URL` = твій прод-домен (напр. `https://www.pidgoretskyymaietok.com`).
4. Передеплой.

Що станеться далі (вже реалізовано в коді):
- При оплаті клієнт іде на захищену сторінку LiqPay, платить карткою.
- LiqPay повертає його на `…/booking?paid=PM-####` і паралельно надсилає серверний
  колбек на `…/api/payment/callback` (з перевіркою підпису).
- Після успіху бронювання позначається оплаченим; при відмові — звільняється.

> Поки оплата у демо, не публікуй сторінку як «справжню оплату». Перемикання на
> бойовий LiqPay — це лише заповнення двох ключів.

## Правила цін і місць

Ціни (грн), залежать від сеансу та будній/вихідний:

| Сеанс | Години | Дорослий Пн–Пт | Дорослий Сб–Нд | Дитина <12 Пн–Пт | Дитина <12 Сб–Нд |
|---|---|---|---|---|---|
| Цілий день | 10:00–20:00 | 500 | 600 | 300 | 400 |
| Вечірнє | 17:00–20:00 | 300 | 350 | 150 | 200 |

- **Дорослий абонемент включає шезлонг** → кількість шезлонгів = кількість дорослих.
- **Діти до 12** — платні, але без власного шезлонга (йдуть «+1» до дорослих).
- **Дітки до 110 см** — безкоштовно, місце не займають.
- 40 шезлонгів (1–40) з фіксованими позиціями навколо басейну.
- Доступність рахується **на дату + сеанс** (одне місце вільне для іншого сеансу/дати).
- Блокування — глобальне (приховує місце від клієнтів незалежно від дати).

Щоб змінити ціни — редагуй `lib/pricing.ts` (одне місце, працює і на клієнті, і на сервері).

## Безпека

- **Ціни рахує сервер.** Клієнт показує суму для зручності, але остаточну суму
  обчислює серверний роут за `lib/pricing.ts`. Підмінити ціну з браузера не вийде.
- **Подвійне бронювання неможливе.** Функція `create_booking` у базі бере advisory-lock
  на слот (дата+сеанс) і перевіряє перетин місць перед вставкою.
- **RLS увімкнено.** Анонімні користувачі не мають прямого доступу до таблиць.
  Перевірка доступності віддає лише номери місць (без імен/телефонів). Усі записи
  клієнта йдуть через серверні роути із service-role ключем.
- **Адмінка під захистом.** `middleware.ts` не пускає неавторизованих на `/admin`,
  а серверна сторінка додатково звіряє email зі списком `ADMIN_EMAILS`.
- **Секрети — тільки на сервері.** `service_role` і приватний ключ LiqPay ніколи не
  потрапляють у клієнтський бандл.
